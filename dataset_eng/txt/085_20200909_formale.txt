The data model describes the management of theatrical activities, including the cataloging of authors, works, theatres, actors and the related ticket office. The Author entity, identified by a unique Code (PK), records personal data such as surname, name, date of birth and date of death. Each Author can be associated with multiple plays in a one-to-many relationship. The Play entity is defined by a Code (PK), Title and Year, and represents the artistic work itself. The Theater entity, identified by a Code (PK), contains logistical information such as Name, Address and Telephone. The actual programming of the shows is managed by the "Opera on billboard" associative entity, whose primary key is composed of the foreign keys (FK) relating to the Theatrical Opera and the Theater in which it is performed (`fk_opera_teatrale_codice`, `fk_teatro_codice`). This entity also includes the Period attribute and implements the one-to-many relationships that link a single Play and a single Theater to multiple schedules. The Actor entity, defined by Code (PK), Last Name and First Name, is linked to "Playlist" via a many-to-many relationship, indicating that an actor can participate in multiple shows and a show can have a cast of multiple actors. Ticket management is based on the "Seat type" and Ticket entities. "Place type" is a registry table with Code (PK) and Description. The Ticket entity is identified by a composite primary key that links a specific "Opera in billboard" (via `fk_opera_in_cartellone_opera` and `fk_opera_in_cartellone_teatro`) to a "Seat type" (`fk_tipo_posto_codice`), and defines the Price. This structure establishes a one-to-many relationship between a schedule and its tickets and between a seat type and its tickets. Finally, the "Reduction Type" entity, with Code (PK) and Description, is linked to Theater by a many-to-many relationship, to specify which reductions are applicable in each theater.
