So basically we have some loose stuff that we need to keep an eye on. First of all, there are the operational headquarters, you know, the offices, each with its own code, address and house number. These jobs, obviously, are located in a certain city, which in turn has a name and an ISTAT code so as not to confuse it with others. The cities are then grouped into regions, these too with their own name and code. Projects are carried out in these offices, each identified by a code and a name. A project takes place in a specific location, but a location can easily host multiple projects at the same time.

Then there are the people who work there, the team members, registered with a serial number and surname. These people are assigned to various projects; one can work on multiple things and a project needs multiple people. When a team member is placed on a project, they also record some kind of allocation data. To understand who is in charge, there is a somewhat convoluted structure that connects people to each other, indicating who is superior and who is inferior in a kind of hierarchy. Each bond of this type is also associated with a description, to specify what type of relationship there is.

The projects, in turn, are broken up into many tasks, which have their own code and name. Each project can have a lot of tasks in it and, perhaps, the same task could be useful for multiple projects. When you link a task and a project, you also throw in a budget estimate. To complete these tasks you need resources, that is, the stuff you need to work, and these also have their own code and name. A task can require multiple resources and a resource can be used for several tasks. Finally, each resource brings with it potential risks, problems that could arise, and these are also cataloged with a code and a name. A risk can be tied to multiple assets, and a resource can have multiple risks associated with it.
