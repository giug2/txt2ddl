So, basically here we're talking about some project-related stuff. Each project has its own name and some sort of explanation, and is always on a single platform, such as web or mobile, there is no confusion. There is also always only one team working on it, so one team can handle multiple projects, but a project is one team's business. These projects are then divided into many sprints, which are periods of work with a duration and an end date that identifies them. Inside the projects there are various functions, each with its name and code. The interesting thing is that the same functionality can be part of several different projects, and obviously a project is made up of many functions. Each functionality, however, is linked to a single customer, who has his own name and surname. A customer can ask for a lot of things, but each request is his and no one else's. Moving on to the teams, each team has a name that distinguishes it and an office where it is located. The people inside the teams are the developers, identified by a serial number, name and surname. The relationship here is a bit strange, it seems like a lot of teams are linked to a single developer, or maybe it's that one developer can only be in one team, it's not super clear. Anyway, then there's this thing called 'Assignment', which is like an assignment. Every team has a bunch of these assignments and every single developer has quite a few of them too, so you understand that an assignment is a specific job given to a person on a certain team. Each assignment, which has its own code, is made using only one specific technology, which has a name and a category. And to complicate things, an assignment can in turn be linked to many other assignments, as if it were a large task divided into many smaller pieces.
