So, basically we have a system that manages a few things, mostly related to restaurants and people. We start from geography, with the Provinces, each with its own acronym that makes it unique and a name. Inside each province you will find a lot of Municipalities, and they too have a code to recognize them and their name. Then, going further down, each municipality has quite a few addresses, which are defined by a street, a number and their specific code.

The heart of everything are the restaurants, each with its own name and code to avoid confusion. What's a little strange is that each restaurant seems to be linked to more than one address. These restaurants are linked to people, whom we recognize by their tax code and whose surname we know. The link is flexible: a restaurant can deal with many people and one person with multiple restaurants, and for each connection there is a kind of "quota" marked, perhaps to understand who owns what.

Moving on to the food, every restaurant has a menu full of dishes. Each dish has a name and its own code. The same dish can be found in different restaurants, but the Cost is specific to that combination, so the price changes depending on where you eat it. Obviously, every dish is made up of ingredients. An ingredient, which always has its own code and name, can be used for a lot of different dishes. And since health is important, we also keep track of which ingredients can cause intolerances or allergies. In practice, one ingredient can be linked to several allergies, and an allergy, which has its own code and name, can be triggered by multiple ingredients.

Finally, there is a part that concerns the bonds between people. Kinship connections can be created between two people. In practice you take two guys, you associate them, and then you specify what type of relationship there is between them using a list of Relationship Types, which has a code and a description to explain whether they are brothers, parents or something else.
