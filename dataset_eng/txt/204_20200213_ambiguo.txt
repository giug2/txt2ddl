So, basically here we have a few things that revolve around Projects. Each project has its own name and a code to identify it, plus a few lines of description to understand what it is about. These projects are made up of a lot of different Modules, and the interesting thing is that a module can be part of multiple projects at the same time, and in each project it is given a certain Priority. Every single module, which has its own name, is always linked to a single Developer, whose name and surname we keep track of; this developer, however, can obviously work on multiple modules.

Returning to the projects, each one is managed by a specific team, which can be recognized by its name and perhaps uses its own work methodology. The code of each project is then in a Repository, which has its own web address, and a project is only in one repository. And obviously, each project has its Releases, which are identified by the date they come out and a version number.

Inside the Teams there are the various Team Members, each with their own personal data. A member is part of only one team, but a team is made up of multiple people. Finally, there is a kind of register called Role Assignment, which is somewhat central stuff: it is used to connect a Team Member with a certain Technology (which has a name and a category) and with the Team itself. Basically, you use this to say who can do what within which workgroup, creating a lot of links to map skills.
