So, basically there is a whole system for managing Projects. Each project has its own identification code, a name and a description to understand what it is about. A project is divided into various Sprints, which are like stages with an end date and a release version; obviously each sprint belongs to only one project. There are teams working on these projects, each with its own name and the date it was born, and a single team can easily deal with multiple projects at the same time.

Then, to understand who does what, we have the people, the Team Members, each identified by a unique serial number, in addition to their name and surname. A person's membership in a team is managed by something called Assignment, which basically acts as a bridge, linking a member's ID number with the team name. It seems that Technologies are also linked to this Assignment, which have their own code and a specific name, perhaps to track who knows how to use what or what tools are needed for that job.

Parallel to all this, there is the figure of the Developer, who has his own unique ID and name. Every developer can get their hands on a lot of different Modules, which are little pieces of software, each with its own code and name. This gets a bit confusing because a single module can be reused in multiple projects and, vice versa, a project can be made up of an infinite number of different modules, so there is a lot of connections between them.
